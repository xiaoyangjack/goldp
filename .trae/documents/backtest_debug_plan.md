# 黄金量化交易系统 - 回测调试与实时运行计划

## 1. 代码逻辑分析

### 1.1 核心模块分析

#### 数据层 (data_fetcher.py)
- **功能**：从akshare获取SGE黄金数据，支持缓存和增量更新
- **逻辑**：
  - 检查缓存数据新鲜度（当天数据）
  - 如有缓存且新鲜，直接使用
  - 否则从akshare获取数据
  - 数据处理：重命名列，计算高低价
  - 增量更新：只添加新数据
  - 数据验证：完整性、一致性、合理性检查
  - 保存数据到CSV
- **问题**：
  - 成交量和成交额为模拟数据
  - 数据验证逻辑可能过于严格

#### 策略层 (web_app.py)
- **功能**：MA过滤策略，生成交易信号
- **逻辑**：
  - 计算MA10和MA30移动平均线
  - 计算MA10-MA30差值
  - 根据差值和阈值生成交易信号（BUY/SELL/WATCH）
  - 根据ATR值判断策略状态（ACTIVE/PAUSE）
- **问题**：
  - 信号生成逻辑较为简单，仅基于MA交叉
  - 没有使用vectorbt进行回测

#### 交易层 (web_app.py)
- **功能**：模拟交易和告警系统
- **逻辑**：
  - SimulatedTrading：管理持仓，计算盈亏，执行交易
  - AlertSystem：发送告警到多个渠道
- **问题**：
  - 交易逻辑固定，每次买入10%资金，卖出50%持仓
  - 没有止损逻辑

#### 展示层 (web_app.py + index.html)
- **功能**：Web界面，展示行情、信号、持仓
- **逻辑**：
  - Flask路由，提供API接口
  - Bootstrap 5响应式设计
  - Plotly图表
- **问题**：
  - 前端代码较为复杂
  - 缺少回测调试界面

### 1.2 数据流分析
1. 数据获取 → 数据验证 → 数据存储
2. 数据读取 → 策略计算 → 信号生成
3. 信号触发 → 交易执行 → 告警推送
4. 数据处理 → 图表生成 → 前端展示

### 1.3 代码逻辑问题
1. **数据层**：
   - 成交量和成交额为模拟数据，可能影响回测准确性
   - 数据验证逻辑可能过于严格，导致数据被拒绝

2. **策略层**：
   - 信号生成逻辑较为简单，仅基于MA交叉
   - 没有使用vectorbt进行回测
   - 策略参数调整后需要重启服务才能生效

3. **交易层**：
   - 交易逻辑固定，每次买入10%资金，卖出50%持仓
   - 没有止损逻辑
   - 告警系统配置复杂

4. **展示层**：
   - 缺少回测调试界面
   - 前端代码较为复杂，维护困难
